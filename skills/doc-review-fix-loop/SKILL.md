## Instructions

[[doc-review-fix]] を繰り返し実行し、全指摘が **CONVERGED** になるまで自動ループする。1パスのレビュー・分類・修正は doc-review-fix 側が担う。本スキルは「収束するまで回す・進捗を見せる・最終レポートを出す」に責務を限定する。

判断基準は一貫して **「読み手が効率よく目的を達成できるか（ISO 9241-11）」**。

### パラメータ

| 引数 | 説明 | デフォルト |
|------|------|-----------|
| 1 | 対象ファイル（レビューするドキュメントのパス） | 直近に作成した成果物 |
| 2 | 最大イテレーション数（安全弁） | 3（収束しなければ打ち切り） |
| 3 | 読み手像「誰に・何を・何のために」 | ドキュメントから読み取る／推定 |

例: `/doc-review-fix-loop docs/manual.md 5` → 最大5回。`/doc-review-fix-loop docs/manual.md` → 既定の最大3回。

### 進捗の可視化（必須）

1イテレーションでサブエージェントを並列起動するため、数分 × 最大N回かかる。沈黙厳禁。ユーザー向けテキストで必ず出す:

1. **着手前**: 「doc-review-fix-loop 開始: <ファイル> を最大Nイテレーション」
2. **各イテレーション開始時**: 「Iter K/N 開始: doc-review-fix 実行中」
3. **各イテレーション完了時**: 「Iter K 完了: critical=x major=y minor=z, Status=CONVERGED/NOT_CONVERGED」
4. **完了時**: 最終レポート表

短文でよい。各イテレーションの詳細は doc-review-fix 側が出すので、ループはサマリだけ蓄積する（全レビュー全文は親コンテキストに溜めない）。

### 設計判断: Agent ラッパーを介さず Skill を直接呼ぶ

doc-review-fix を起動するのに general-purpose Agent でラップしない。ラッパーは文脈分離のためだけに存在し、起動のたびに全システムプロンプト・全 Skill 定義を再ロードして遅い。文脈分離は doc-review-fix が**内部で並列起動するレビュー用サブエージェント**で十分確保される。結果は結果ファイル経由で親に伝わる。

そこで本スキルは **main session から [[doc-review-fix]] を Skill ツールで直接呼ぶ**。

### 実行手順

イテレーション・カウンタ **K は 1 から開始**し、手順0〜5を1巡するごとに 1 巡＝1イテレーションとして数える。N は最大イテレーション数（パラメータ2、既定3）。手順5で「K < N かつ未収束なら K を +1 して手順0へ」「K = N で未収束なら打ち切り」と判定する（N回目を実行し終えてから打ち切る）。

#### 0. 前回結果ファイルの削除

各イテレーション開始前に、前回の結果ファイルを削除する（残存による誤判定を防ぐ）。doc-review-fix と同じ結果ファイルパスを使う（既定は一時ディレクトリの `doc-review-fix-result.md`）:

```
rm -f <結果ファイルパス>
```

#### 1. doc-review-fix の呼び出し

Skill ツールで [[doc-review-fix]] を呼ぶ（対象ファイル・読み手像・結果ファイルパスを引数で渡す）。doc-review-fix 内部で白箱1＋読み手N名のレビュアー・サブエージェントが並列起動し、レビュー・分類・自動修正・結果出力までを1パスで完遂する。

#### 2. 結果ファイル検証

doc-review-fix 完了後、結果ファイルを Read で読む。ファイルが存在しない、または `対応サマリ` テーブル・`収束判定` セクションを含まない場合は、正しく実行されなかったとみなし **NOT_CONVERGED** とする。

#### 3. 収束判定

結果ファイルの `対応サマリ` テーブルと `収束判定` を見て、このイテレーションの**判定ラベルだけ**を決める。**ここでは終了・分岐しない**——終了するかどうかは手順4（サマリ蓄積）を必ず済ませてから手順5で判断する。収束していてもこのイテレーションの行を最終レポート表に残すため、この順序を厳守する。

判定は一貫して **「このパスが新たに何を検出したか」（対応サマリの "検出" 列）** で行う。"修正済み" で残置 0 でも、検出された以上そのパスは指摘を出している＝デグレ確認のため次パスが要る。**「検出 = 0」になって初めて収束**とする（要確認のみの例外は下記）。"要確認で残置" は捏造リスクで自動修正できず残るもので、これは収束を妨げない。

| 条件（対応サマリの "検出" 列で判定） | 判定ラベル |
|------|------|
| critical・major の**検出**が両方 0 | **CONVERGED** |
| このパスで検出された critical/major のうち「要確認で残置（捏造リスク）」を除くと 0 件（＝非要確認の検出が 0、要確認のみ） | **CONVERGED**（要確認を明記済みであること） |
| 非要確認の critical または major が1件でも**検出**された | **NOT_CONVERGED** |
| ファイル不在・形式不正 | **NOT_CONVERGED** |

**重要**: doc-review-fix が `CONVERGED` と自己申告していても、要確認以外の critical/major が**検出**されていれば NOT_CONVERGED とする。修正を行った以上デグレの可能性があるため、収束は対応サマリの検出数で裏取りする。

#### 4. サマリ蓄積

結果ファイルの `対応サマリ` から critical / major / minor の検出数を読み取り、手順3の判定ラベルと組にして **1行サマリ（`Iter K | critical | major | minor | 判定ラベル`）** として記録する。最終レポート表の1行はこれをそのまま使う。

打ち切り（手順5の `K = N` 分岐）に備え、**最後のイテレーションの結果ファイルは削除せず残す**（手順0の削除は次イテレーション開始時のみ）。打ち切り時はこの結果ファイル本文（`主な修正` や各指摘）から、**最後のパスで検出された要確認以外の critical/major** の個別指摘を拾って最終レポートに列挙する。最後の結果ファイルが不在・形式不正で打ち切った場合は、列挙の代わりに「最終パスが正常な結果を出せなかった」旨を記す。

#### 5. ループ継続 / 終了判断

手順4でこのイテレーションの行を記録し終えてから、ここで初めて終了するか継続するかを決める（終了は必ずこの手順に一元化する）:

- 手順3の判定ラベルが **CONVERGED** → ループを抜けて最終レポートへ。
- **NOT_CONVERGED** かつ `K < N` → K を +1 して手順 0 に戻り次イテレーションへ。
- **NOT_CONVERGED** かつ `K = N`（最大に到達） → 打ち切り。未解決の指摘（最後のパスで検出された要確認以外の critical/major）を最終レポートに列挙して終える。

**捏造リスク（要確認）だけが残って収束しないループは回さない**——doc-review-fix がそれを収束扱い（CONVERGED）にするので、本ループは無限化しない。

### 最終レポート

```
# doc-review-fix-loop 結果: <ファイル>

| Iter | critical | major | minor | 結果          |
| ---- | -------- | ----- | ----- | ------------- |
| 1    | 2        | 3     | 4     | NOT_CONVERGED |
| 2    | 1        | 0     | 1     | NOT_CONVERGED |
| 3    | 0        | 0     | 0     | CONVERGED     |

**最終結果: CONVERGED**（または NOT_CONVERGED: 最大Nイテレーションで打ち切り）

## 未解決の指摘（NOT_CONVERGED で打ち切った場合のみ）
- <最後のパスで検出された要確認以外の critical/major: 観点・箇所・内容。最終イテレーションの結果ファイル本文から転記。結果ファイルが不在・形式不正だった場合は「最終パスが正常な結果を出せなかった」旨を記す>

## 要確認で残した不足（あれば）
- <何が足りないか / 誰に確認すべきか>
```

`Iter` 表の各行は手順4で記録した1行サマリ（件数＋判定ラベル）をそのまま並べる。「未解決の指摘」節は CONVERGED 時は省略してよい。

---

### 注意事項

- レビュー用サブエージェントは doc-review-fix の Phase 1 内で並列起動され、それぞれ独立コンテキストで動く。親（このループ）には doc-review-fix 本文と各イテレーションのサマリだけが蓄積される。
- 修正適用（Phase 3）は doc-review-fix が main session で行うため、ユーザーは進行を直接観察できる。
- サブエージェントを dispatch できない環境では、doc-review-fix が白箱＋劣化版自己レビューに縮退する（その旨が結果ファイルに明記される）。ループはそのまま回せる。
- このループ全体が [[clear-document-writing]] の Step 6 の実体。作成（Step 1〜5）の直後に呼ぶことで、作成からレビュー修正までが一貫する。
